# 운영 규칙

## 런타임 모드

### `standard`
시간이 충분할 때 사용.

- 신중하게 분류
- 모호성이 있을 때 후보 2~3개 생성
- 모든 제출 전 검증
- 비자명한 문제에서 `Critic` 사용

### `compressed`
시간이 촉박하지만 검증 한 번은 가능할 때 사용.

- 지배적인 플레이북 하나를 빠르게 결정
- 후보 1~2개만 생성
- 짧은 Verifier 체크리스트 사용
- 위험이 높지 않으면 `Critic` 생략

### `recovery`
제출 실패 또는 출력 충돌 후 사용.

- 문제를 처음부터 다시 기술
- 실패 원인 분리
- 주요 변수 하나만 변경
- 반복하지 말아야 할 것을 문서화

## 모델 분기 기준

단일 모델 유지 조건:
- 과제가 단순하고 범위가 명확할 때
- 답 형식이 결정론적일 때
- 실패 비용이 낮을 때

다중 모델로 분기 조건:
- OCR 또는 비전 모호성이 높을 때
- 여러 추론 경로가 다른 결론에 도달할 때
- 대안 가설이 과제에 유리할 때
- 현재 모델이 자신 있게 틀리거나 불안정할 때

## 코드 사용 기준

허용하되 선택적으로 사용.

사용 가능한 경우:
- OCR 후처리
- 반복 포맷팅
- 후보 순위 결정 또는 비교
- 작은 탐색 공간의 브루트포스 확인
- 빠른 파싱, 계산, 변환

사용하지 말아야 할 경우:
- 모델이 더 빠르게 할 수 있는 막연한 브레인스토밍
- 설정 비용이 절감 시간보다 큰 무거운 파이프라인
- 문제가 이해되기 전의 취약한 자동화

## Human Checkpoint 기준

다음 상황에서 Human Checkpoint 열기:
- 시각적 증거가 모호할 때
- OCR 출력에 명백한 오염이 있을 때
- 후보 답안이 비슷하지만 동일하지 않을 때
- 최종 답이 순수 추출이 아닌 판단을 요구할 때

체크포인트는 다음만 결정:
- 유지할 것
- 버릴 것
- 다음에 제출할 것

## 검증 규칙

모든 진지한 후보는 다음을 확인해야 함:

- 정확한 출력 형식
- 제약조건 충족
- 단위 및 범위 일관성
- 숨겨진 엣지 케이스
- 출처 자료와의 모순

## Submission Loop 규칙

복수 제출 또는 점수 피드백이 있을 때:

- 점수를 높인 것은 유지
- 한 번에 주요 변수 하나만 변경
- 하나의 재시도에 새 아이디어 세 가지를 혼합하지 않음
- 단일 답 추측이 아닌 순위가 매겨진 후보 목록 유지

## 중단 규칙

다음 상황에서 복잡도 확장 중단:

- 제출 준비된 후보가 검증을 통과할 때
- 다음 분기의 기대값이 낮을 때
- 코드 경로가 문제보다 무거워질 때
- 모델 출력이 증거 없이 발산할 때
